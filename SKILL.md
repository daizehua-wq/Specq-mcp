# SpecQ 攻单情报包 — 电子化学品销售 Skill

## 适用场景

电子化学品销售 / 售前工程师见客户前，快速获得结构化攻单情报包。

## 触发条件

用户提供以下**任一组信息**即可触发：

| 触发模式 | 示例 |
|---|---|
| 三字段 | "帮我做XX电镀液的情报包，客户深南电路，他们关注粗糙度" |
| 查历史 | "深南电路之前聊了什么？" |
| 反馈 | "深南成交了" / "丢了，他们嫌价格贵" |

## 依赖 MCP Server

| Tool | 输入 | 输出 |
|---|---|---|
| `specq_generate_intel` | product, application, scenario | 八模块情报包 Markdown |
| `specq_extract_insights` | customer_id（可选）, limit | 结构化洞察 JSON |
| `specq_log_visit` | customer_id, content, visit_date, visit_type | 拜访记录 ID |
| `specq_feedback` | product, application, outcome, lesson, accuracy_notes | 反馈记录 |

---

## 工作流

```
用户输入
  ↓
提取 product / application / scenario（三字段）
  ↓
尝试 extract_insights（查该客户历史拜访/丢单记录）
  ├─ 有暗数据 → 注入 scenario → 继续
  └─ 无暗数据 → 继续（模块4/5/6 标注[经验推断]）
  ↓
specq_generate_intel(product, application, enhanced_scenario)
  ↓
输出八模块情报包（Markdown）
  ↓ ⚠️ 重要：必须提醒反馈闭环
"见完客户后告诉我结果（成交/丢单/继续跟进），我帮你记录，情报包会越用越准。"
```

## 分支场景

### 场景 A：新客户（无历史数据）
- 直接调 `specq_generate_intel`
- 模块 4/5/6 标注 `[经验推断，暂无销售数据支撑]`

### 场景 B：老客户（有拜访记录）
- 先调 `specq_extract_insights` → 提取关注指标、丢单原因
- 注入到 scenario 参数中
- 情报包模块 4/5/6 有真实数据来源标注

### 场景 C：仅查历史
- 用户问历史 → `specq_extract_insights` → 结构化展示
- 展示后追问："要基于这些信息生成情报包吗？"

### 场景 D：成交反馈
- 用户报告结果 → `specq_feedback` → 记录闭环
- outcome: won（成交）/ lost（丢单）/ follow_up（跟进中）

## 输出规格

情报包按八模块输出，每个模块标注数据来源：

| 模块 | 内容 | 数据来源 |
|---|---|---|
| 1. 产品概览 | 产品定义、核心功能、适用工艺段 | 公开资料 + knowledge.db |
| 2. 技术指标对比 | 关键参数 vs 竞品/行业标准 | 公开资料 + 暗数据 |
| 3. 竞品格局 | 主要竞品、市占、差异化 | 公开资料 |
| 4. 客户关注指标 | 该客户/行业重点技术指标 | 暗数据 / [经验推断] |
| 5. 切入机会 | 当前该客户的切入窗口 | 暗数据 / [经验推断] |
| 6. 导入障碍 | 历史丢单原因、技术壁垒 | 暗数据 / [经验推断] |
| 7. 行动建议 | 拜访话术、演示重点、报价策略 | LLM 综合生成 |
| 8. 参考来源 | 各模块数据来源 + 置信度 | 自动标注 |

---

## 重要规则

1. **暗数据优先**：有历史拜访/丢单记录时必须注入，不得跳过
2. **来源透明**：每模块必须标注数据来源，暗数据标注 `[来自销售拜访记录]`，推测标注 `[经验推断]`
3. **反馈必提醒**：每次输出情报包后必须提醒用户反馈结果，不得遗漏
4. **参数不足时**：如果三字段提取不全，主动追问缺失字段，不猜测
5. **保密**：情报包内容不包含客户真实姓名/可识别信息（脱敏为行业标签）

---

## MVP 边界

**做**：
- 场景 A/B/C/D 全部覆盖
- 纯 Markdown 输出

**不做**：
- 多轮对话深挖需求（单轮优先）
- 自动推送情报包到飞书/邮件
- 竞品实时价格
- 可视化图表
